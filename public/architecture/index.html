2. Part A — From ISA to single-cycle microarchitecture (Lecture 4)

2.1 ISA vs microarchitecture

The Instruction Set Architecture (ISA) is the contract between the programmer and the machine. It defines how instructions are encoded in binary, what each instruction means, which registers exist and how they are named, and how memory is addressed. The ISA says nothing about how the hardware carries out those instructions—only what the result must be.

The microarchitecture is the actual hardware implementation of that contract. It has two cooperating parts: the datapath—the physical components that move and transform data (ALU, register file, instruction memory, data memory, adders, and multiplexers)—and control—the combinational logic that reads the opcode and funct fields of each instruction and asserts the control signals (RegWrite, MemRead, MemWrite, ALUSrc, etc.) that configure the datapath for that particular instruction.

Lecture 4’s baseline design is single-cycle (non-pipelined): every instruction, regardless of type, completes in exactly one long clock cycle, from instruction fetch all the way through write-back. Because all instructions must share the same clock period, the period must be long enough to satisfy the slowest instruction. The slides identify the critical-path instruction as load word (lw), since it passes through every functional unit in sequence: I-memory → register file read → ALU (address calculation) → D-memory read → register file write.

Textbook alignment: Hennessy & Patterson, Computer Organization and Design (or COD), Chapter III, sections III.I–III.IV (addressing, single-cycle datapath, control, critical path).

2.3 Instruction life-cycle (logical phases in one long cycle)

Even though every instruction occupies exactly one clock cycle in the single-cycle model, that cycle can be broken into six logical phases that correspond to the hardware resources used in sequence:

  1. Fetch: the current PC is sent to instruction memory, which returns the 32-bit instruction word. The PC adder simultaneously computes PC+4 for the default next instruction.
  2. Decode: the instruction word is parsed—opcode, register specifiers (rs, rt, rd), the funct field for R-type, and the immediate for I-type. The register file is read with rs and rt as addresses, producing the two source operand values. Immediates are sign-extended here.
  3. Execute: for R-type instructions, the ALU applies the operation indicated by funct. For loads and stores, the ALU computes the effective address as R[rs] + sign_ext(imm). For branch instructions, the ALU subtracts the two source registers (for beq) and asserts the Zero flag if they are equal; in parallel, the branch target address is computed as (PC+4) + (sign-extended offset × 4).
  4. Memory access (loads and stores only): a load sends the effective address to data memory and reads back a value; a store writes the value in R[rt] to that address. Instructions that are not loads or stores effectively skip this phase (the memory is idle).
  5. Write-back (if a result must be saved): R-type instructions write the ALU result to R[rd]; lw writes the loaded value to R[rt]. Stores and branches do not write a register.
  6. PC update: the PC is set to PC+4 for the sequential next instruction, to the branch target if a taken branch was detected, or to the jump target for j/jal.

Design choice: the full datapath uses a Harvard-style split between instruction memory and data memory (labelled I$ and D$ in real systems). This is necessary because, in a single cycle, the CPU needs to read an instruction and (for loads/stores) access data memory at the same time—a single unified memory with one port could not service both requests simultaneously.

2.4 Logic design vocabulary

The datapath is built from two classes of hardware elements. Combinational elements produce an output that is a pure function of their current inputs—there is no stored state. Examples used in the MIPS datapath include AND gates, the 2:1 multiplexer (Y = S ? I1 : I0), the adder that computes PC+4, and the ALU itself. Because they have no memory, combinational elements settle to a new output as soon as their inputs change, subject only to propagation delay.

Sequential elements do hold state. The fundamental building block is the D flip-flop, which captures the value on its D input at the rising (or falling) clock edge and holds it stable on output Q until the next edge. The Program Counter (PC) is the most visible sequential element in the single-cycle design—it stores the address of the current instruction and is updated once per cycle. The register file is a bank of flip-flops. The rule that defines when signals must be stable—setup time before the clock edge and hold time after—is called the clocking methodology, and it ensures that the longest combinational path between any two registers fits within one clock period.

Multiplexers are the key decision-making elements. They select between alternative data paths based on a control signal. For example, one mux selects whether the ALU’s second input comes from R[rt] (for R-type) or from the sign-extended immediate (for loads, stores, and I-type ALU ops); another mux selects whether the write-back value comes from the ALU result or from data memory (for loads).

2.7 Performance: critical path and improving performance

In a synchronous digital circuit, the clock period is set by the longest combinational path from any flip-flop output to any flip-flop input—because all state elements must capture stable values on every clock edge. The slides state this directly: “Longest delay determines clock period.”

For the single-cycle MIPS datapath, the critical path runs through the load instruction: I-memory (fetch) → register file read → ALU (address add) → D-memory read → register file write. An R-type instruction is actually faster because it skips the data memory access, but since every instruction must complete within the same clock period, the entire machine’s clock is forced to accommodate the slowest case—the load. The clock cannot be shortened just because the current instruction happens to be an R-type.

The slides pose the rhetorical question “Vary clk period for different instructions?” to highlight that a conventional single-edge synchronous design cannot do this—all instructions must share one fixed period. This inflexibility is a core motivation for moving to pipelining, which replaces that one long cycle with several shorter, overlapping stages. Other approaches to improving performance in later material include a faster memory hierarchy (caches) and architectural changes that reduce the critical path length in each stage.

3. Part B — Pipelining I: motivation, five-stage model, hazards overview (Lecture 5)

Textbook alignment: Hennessy & Patterson, Chapter IV; IV.IV and IV.V (or equivalent in your edition) cover pipelining foundations and hazards / introductory branch material.

3.1 Brainiac vs speed daemon and complexity-effective design

Lecture 5 opens with a conceptual tension that runs through the rest of the course. A Brainiac processor is one that sacrifices simplicity for intelligence—it adds wide-issue logic, deep out-of-order scheduling, and elaborate bypass networks to maximize the number of instructions completed per cycle (IPC). A Speed Daemon processor takes the opposite trade: it keeps each pipeline stage as simple as possible so the critical path through each stage is short and the clock can run faster (smaller T_c). Neither approach is free: the Brainiac’s extra hardware makes each clock cycle longer, while the Speed Daemon may finish fewer useful instructions per cycle.

The deck includes an excerpt from Complexity-Effective Superscalar Processors (Palacharla, Jouppi, Smith), which argues that hardware complexity ultimately shows up as logic delay on critical paths, and that the slowest stage determines the processor clock. Wide-issue machines in particular stress the issue-window wakeup/select logic and the operand bypass network; as technology scales, wire delay (not gate delay) can dominate these paths and limit how fast the clock can run.

The course’s bottom line for this slide: wall-clock execution time depends on both CPI and T_c. Pipelining—the main topic of Lectures 5–7—primarily improves throughput by overlapping instruction phases, which reduces the effective per-instruction clock cost. It does not make any single instruction complete faster end-to-end; it just allows many instructions to be in-flight simultaneously.

3.2 Pipelining definition and laundry analogy

Pipelining achieves parallelism in time by overlapping the execution phases of multiple instructions simultaneously. Rather than waiting for one instruction to finish all five stages before starting the next, a pipelined processor starts a new instruction every cycle, so at any given moment, up to five instructions are in-flight—each in a different stage.

The classic analogy in the slides is a laundry pipeline: if you have four loads of laundry and four steps (wash, dry, fold, put away), you dramatically cut total time by starting the second load in the washer while the first is in the dryer, and so on. The speedup from the slides for a small finite batch is about 8/3.5 ≈ 2.3×. In the steady state—when the pipeline stays full—speedup approaches the number of stages (e.g. ~4× for a 4-stage pipe) when stages are perfectly balanced. The five-stage MIPS pipeline can ideally approach 5× throughput improvement over single-cycle for the same per-stage delay, though hazards and imbalanced stage latencies reduce this in practice.

The single-cycle design must clock at T_c,single = 800 ps because lw is the slowest instruction. In the pipelined design, the clock period only needs to cover the slowest stage, which is T_c,pipe = 200 ps. This gives an ideal throughput speedup of 800/200 = 4× in this particular model (not quite 5× because the register stages are only 100 ps and the pipeline clock is bottlenecked by the 200 ps stages).

It is important to distinguish latency from throughput. A single instruction still takes all five stages to complete, so its end-to-end latency is still roughly 5 × T_c,pipe. What improves is throughput: once the pipeline is full, one instruction completes every cycle. This is only achievable when (1) stages are balanced in delay (if one stage is far slower, it pins T_c and everyone waits for it), and (2) there is enough work to keep the pipe full. In practice, hazards (§3.6), latch overhead, and memory effects reduce real throughput below the ideal, and these topics are developed in Parts C and D.

3.5 ISA features that ease pipelining (MIPS vs x86 on slides)

MIPS was designed with pipelining in mind, and several of its ISA choices make the five-stage pipeline much simpler to implement. All MIPS instructions are exactly 32 bits wide, so the fetch stage always knows how many bytes to read. This contrasts sharply with x86, whose instructions range from 1 to 17 bytes; the x86 frontend must include complex variable-length decode logic before it can even begin to dispatch an instruction. MIPS also uses few, regular instruction formats (R, I, and J), so that register fields appear at the same bit positions across instruction types and the decode stage can read the register file before it even knows what the instruction is going to do.

The load/store architecture means that memory addresses are computed in the EX stage and the actual memory access happens in MEM—these two operations are cleanly separated into different pipeline stages with no overlap. Finally, MIPS requires aligned operands (word accesses must be at 4-byte-aligned addresses), so every memory access fits into a single MEM stage with no multi-cycle complications.

3.6 Three hazard types (MIPS stories)

Three categories of pipeline hazard prevent the ideal one-instruction-per-cycle throughput, and each has a different source and solution:

  1. Structural: shared resource; classic example single single-ported memory for IF and MEM → IF stalls, bubble. Mitigation: Harvard split (I-mem / D-mem).
  2. Data (RAW): an instruction depends on a value produced by an earlier instruction that is not yet visible at the point of use (read-after-write). Example (slides):
add  $s0, $t0, $t1
sub  $t2, $s0, $t3

sub reads $s0 in ID while add may not write $s0 until WB — without forwarding, sub sees a stale value.

  • Forwarding (bypassing): route the result from EX/MEM or MEM/WB to ALU inputs via extra muxes and a forwarding unit (Lect06–07).
  • Load–use: still needs a 1-cycle stall (detection in ID) — load data does not exist until after MEM, one cycle too late for the immediately following instruction at EX.
  • Code scheduling: reorder independent instructions between load and consumer to hide the stall. Example from slides:
A = B + E;
C = B + F;

3.7 Single-cycle vs pipelined (visual message)

Both the single-cycle and pipelined designs use the same functional components—I-memory, register file, ALU, D-memory, muxes, adders. The key structural difference is that the pipelined version inserts pipeline registers (latches) between each pair of stages (IF/ID, ID/EX, EX/MEM, MEM/WB). These latches capture the outputs of each stage at the clock edge and hold them stable while the next stage operates. This staging allows different instructions to occupy different stages simultaneously. The flip side is that the design must now include hazard detection and forwarding logic to handle situations where an instruction in a later stage needs a value that an earlier instruction in the pipeline has not yet produced.

4.2 Pipelined vs single-cycle

The pipelined datapath uses the same functional blocks as the single-cycle design (I-mem, PC+4 adder, register file, ALU, D-mem, sign-extender, muxes, and control logic). The difference is that pipeline registers are inserted between stages, and the control signals for each instruction must travel with the instruction through the pipe. When the decode stage determines the control values for an instruction (RegWrite, MemRead, ALUSrc, etc.), those values are latched into ID/EX along with the register data. They then travel forward through EX/MEM and MEM/WB, arriving at each stage just in time to control the appropriate hardware.

Because MIPS uses fixed bit positions for register fields across all instruction types, the decode stage can split the instruction word into fields purely by wiring—no complex variable-length parsing is needed. The decode table maps opcodes to control signals and feeds the results into the ID/EX latch for downstream stages to use.

4.3 Pipeline register contents (conceptual)

Each pipeline register holds all the information that downstream stages will need to complete the current instruction. Conceptually:

  • IF/ID holds the fetched instruction word and the computed PC+4, which is needed later for branch target calculation.
  • ID/EX holds the two register read values (Reg[rs] and Reg[rt]), the sign-extended immediate, the register specifiers (so forwarding logic can check for matches), and all the control signals for EX through WB.
  • EX/MEM holds the ALU result (which is either a computed value or a memory address), the store data (Reg[rt], for store instructions), the destination register Rd after the RegDst mux has selected between rd and rt, and the remaining control signals for MEM and WB.
  • MEM/WB holds either the data read from memory (for loads) or the ALU result passed through, along with the destination register Rd and the write-back control signals (RegWrite, MemtoReg).

Note: branch resolution in ID and forwarding to the ID stage are handled in Lecture 7. Lecture 6 focuses specifically on ALU forwarding to EX and the EX/MEM priority rule.

4.4 Load through the pipeline

Tracing a lw instruction through the five stages is the clearest way to understand why the load-use hazard exists. The stages proceed as follows:

  1. IF: The lw instruction is fetched from I-memory.
  2. ID: The instruction is decoded. The base register is read from the register file, the 16-bit offset is sign-extended to 32 bits, and the control signals MemRead, MemtoReg, RegWrite, and ALUSrc are asserted and latched into ID/EX.
  3. EX: The ALU adds the base register value and the sign-extended offset to produce the memory address. Critically, the actual load data is not available yet—only the address has been computed. The result is latched into EX/MEM.
  4. MEM: The computed address is sent to data memory, which returns the loaded word. This value is latched into MEM/WB. This is the first point at which the loaded data exists anywhere in the pipeline.
  5. WB: The MemtoReg mux selects the data from memory (rather than the ALU result) and writes it back to destination register rt.

The key insight is that the loaded value is not available until after the MEM stage completes. If the very next instruction needs that value in its EX stage, it arrives one cycle too late even with forwarding—this is the load-use hazard, and it requires a one-cycle stall (see §5.3).

4.5 RAW and forwarding to EX

A Read-After-Write (RAW) hazard occurs when an instruction needs a register value that a not-yet-completed earlier instruction is supposed to produce. Without any mitigation, the consumer would read a stale (pre-write) value from the register file. The naive fix—stalling until the producer reaches WB—wastes one or two cycles on every dependency.

Forwarding (also called bypassing) avoids most of these stalls by routing the result of a producer instruction directly from a pipeline register to the ALU input of the consumer, without waiting for the value to be written to the register file and read back. Two forwarding paths are provided:

  • EX/MEM → EX: forwards the result of the instruction that just completed EX (i.e., it is one step ahead of the consumer). This handles the distance-1 dependency case.
  • MEM/WB → EX: forwards the result of the instruction two steps ahead of the consumer. This handles the distance-2 dependency case.

In both cases, forwarding is triggered when three conditions hold simultaneously: (1) the producer has RegWrite = 1, (2) the producer’s destination register is not $0 (writes to $0 are discarded by definition), and (3) the producer’s destination register number matches the consumer’s source register (rs or rt in ID/EX). Store instructions introduce a second forwarding case: the store’s data operand may itself come from MEM/WB and needs to be forwarded to the MEM stage, not just to EX.

4.6 Double hazard and revised MEM condition

The most subtle forwarding scenario arises when two earlier instructions have both written to the same destination register and are simultaneously in-flight—one in EX/MEM and one in MEM/WB. The consumer in EX needs a value, both pipeline registers “claim” to hold the right value for that register name, and the forwarding mux must decide which one to use.

The answer follows from program order: the instruction in EX/MEM is younger (more recently issued) than the one in MEM/WB. In the sequence of instructions, the younger write supersedes the older one—so the value from EX/MEM is the correct, most up-to-date value. The MEM/WB value is stale for that register.

The priority rule is therefore: check EX/MEM first; fall back to MEM/WB only if EX/MEM does not match. Expressed as a priority-encoded mux selection:

  1. If EX/MEM.RegWrite = 1 and EX/MEM.Rd ≠ $0 and EX/MEM.Rd matches the needed source → forward from EX/MEM.
  2. Else if MEM/WB.RegWrite = 1 and MEM/WB.Rd ≠ $0 and MEM/WB.Rd matches and the EX/MEM condition above did NOT also match → forward from MEM/WB.
  3. Otherwise → use the register file value latched in ID/EX.

The negation of the EX/MEM condition in step 2 is what makes the MEM/WB path “revised”—without it, the mux might select the older, stale value when both registers simultaneously match. A special case: never forward from a write to $0, since $0 is hardwired to zero and any write to it is discarded. Finally, note that the rs and rt source operands are checked independently—one might forward from EX/MEM and the other from MEM/WB in the same cycle (the double-forwarding case explored in the discussion handout).

5. Part D — Pipelining III: load–use, control, prediction, BTB (Lecture 7)

Focus: Pipelining III (load–use, control, prediction). The forwarding matrix discussion in §7 F.5 matches the same forward-to-ID and distance ideas developed here in §5.6–5.8.

5.1 Default model (reiterated for Lect07 focus)

IF, ID, EX, MEM, WB; branches in ID; no prediction unless specified.

5.2 Forwarding on datapath (recap)

The forwarding unit sits in front of the two ALU inputs in the EX stage and selects the correct operand source using the priority-encoded conditions from §4.6. An additional convention used in the course model is the WB half-cycle rule: the register file write happens in the first half of the WB stage’s cycle while the register file read happens in the second half of the same cycle, so a value being written in WB can be “bypassed” to an instruction reading in ID within the same cycle. This effectively eliminates the distance-2 stall for R-type instructions without needing a dedicated pipeline latch path.

However, forwarding alone cannot eliminate the load-use hazard. When a load is followed immediately by an instruction that needs the loaded value, the data physically does not exist until after MEM—which is one cycle later than EX needs it. No amount of wiring can forward a value before it has been computed. This case requires a one-cycle stall, covered in §5.3. Control hazards similarly cannot be resolved by forwarding; they require flushing incorrectly fetched instructions, which is separate from data bypassing.

5.3 Load–use hazard

The load-use hazard is the one situation where forwarding cannot help on its own. When a lw is immediately followed by an instruction that reads the loaded register, the consumer instruction reaches its EX stage in the same cycle that the load is in its MEM stage—but the loaded data won’t come out of data memory until the end of MEM. The consumer needs the value at the start of EX, which is one cycle too early.

The solution is to insert exactly one stall (bubble) cycle. The hardware detects this situation while the load is in ID/EX and the consumer is in IF/ID, using the condition:

ID/EX.MemRead = 1
AND (ID/EX.RegisterRt = IF/ID.RegisterRs
  OR ID/EX.RegisterRt = IF/ID.RegisterRt)

This checks whether (a) the current instruction in EX is a load (MemRead asserted), and (b) the load’s destination register (RegisterRt for lw) matches one of the source registers the next instruction is about to read. If both conditions are true, the consumer’s data won’t be ready in time.

When the stall fires, the hardware freezes the front of the pipeline (holds PC and IF/ID unchanged, so the same instructions will be re-processed next cycle) and injects a no-op bubble into ID/EX (all control signals zeroed). After one stall cycle, the load has completed MEM and its result is in MEM/WB, from which it can be forwarded normally to the consumer’s EX stage. As the slides put it: “1-cycle stall allows MEM to finish read for load; data can be subsequently forwarded to EX (consumer).”

5.4 Stall implementation (control-signal view)

Inserting a stall cycle requires coordinated changes to three parts of the pipeline control:

  1. Force ID/EX to a no-op: set all control bits in the ID/EX latch to zero. This prevents any spurious register write, memory access, or other side effect from the instruction that was in ID when the hazard was detected.
  2. The bubble propagates normally through EX, MEM, and WB as a harmless no-op, occupying pipeline stages without doing any work.
  3. Hold PC and IF/ID frozen: by disabling the write enables on the PC register and the IF/ID latch, the pipeline stalls the front end for one cycle. The instruction that was in IF is fetched again next cycle, and the instruction in ID (the consumer) is re-decoded next cycle with the correct, now-available forwarded value.

The net effect is that the front of the pipeline freezes while the back drains, creating exactly the one-cycle gap needed for the load’s data to become available for forwarding.

5.5 Integrated datapath: hazard + forwarding

The complete pipelined datapath integrates two new hardware units beyond the base five-stage design. The forwarding unit in EX monitors the EX/MEM and MEM/WB registers, compares their destination register fields against the source operands in ID/EX, and drives the mux select signals in front of each ALU input. The hazard detection unit in ID checks for the load-use condition and, when it fires, drives the write-enable signals of the PC and IF/ID latch low (holding them frozen) while simultaneously inserting a zero-control bubble into ID/EX. The ID stage can also receive bypass paths from EX/MEM or MEM/WB for branch operand comparisons (developed in §5.8).

5.6 “Forward to MEM?” (design Q from slides / discussion handout)

The discussion handout poses the question: can we forward to the MEM stage, not just to EX? Two cases arise in the forwarding matrix:

EX → MEM as a consumer: For ordinary register RAW hazards, this is not a useful forward path. By the time an instruction is in MEM, it is no longer computing with register operands—it is reading or writing memory using an address that was already computed in EX. The forwarding mux at the EX stage input is the correct place to intercept register dependencies before that address computation happens. Forwarding into MEM “as a register consumer” would violate program order.

MEM → MEM: This path can exist in principle—for example, if a store instruction needs to write a value that a load (one cycle earlier) has just read. However, the handout and lecture conclude that the benefit is typically low: if earlier R-type forwarding (EX→EX or MEM→EX) has already handled the dependency for the store’s data register, the MEM→MEM path is redundant. The design question is whether the added hardware complexity of a second MEM bypass network pays for itself; the course’s answer is that in most cases it does not, and you should assess the benefit case-by-case rather than assuming a full second MEM forward network is always present.

5.7 Data vs control hazards (slide definitions)

Data hazards arise when an instruction needs a value that an earlier instruction has not yet written. The standard responses are forwarding (to avoid stalls entirely for most cases) and stalling (for the load-use case). The key property of data hazards is that the earlier instruction (the producer) is the one that the hardware tracks—it is the producer’s pipeline stage that determines what forwarding path is available.

Control hazards arise when the processor has fetched instructions past a branch before knowing whether the branch is taken. If the branch turns out to be taken, the fetched fall-through instructions are on the wrong path and must be flushed before they can write state. The response to a control hazard is always flush (squash), not stalling and forwarding—the wrong-path instructions must be discarded entirely.

In the course’s default model, branches are resolved in the ID stage using a dedicated comparator and a branch target adder in ID. This means only the instruction that was already in IF (one instruction past the branch) must be flushed on a taken branch—a one-instruction penalty. If the branch were resolved later (e.g. in EX or MEM), more instructions would be on the wrong path and the penalty would be higher.

The slides illustrate this with the following sequence (addresses in bytes):

36:  sub  $10, $4, $8
40:  beq  $1, $3, 7
44:  and  $12, $2, $5
48:  or   $13, $2, $6
52:  add  $14, $4, $2
56:  slt  $15, $6, $7
...
72:  lw   $4, 50($7)

5.9 Dynamic branch prediction

As pipelines get deeper, the number of instructions that get fetched on the wrong path before a branch resolves increases—and so does the penalty of flushing them. At some point, the cost of always stalling or always flushing becomes unacceptable, and the processor needs to predict branch outcomes before they are known.

A Branch History Table (BHT) stores the recent outcome (taken / not taken) of branches indexed by their PC (or a hash of it). When the processor fetches an instruction whose PC matches an entry in the BHT, it uses that entry’s stored prediction to decide which path to fetch next, without waiting for the branch to actually resolve. If the prediction turns out to be correct, execution continues with no penalty. If wrong, the incorrectly fetched instructions are flushed and fetch restarts at the correct target.

5.10 Two-bit saturating counter (bi-modal)

A single-bit predictor flips its prediction after every misprediction—so a branch that is taken 99% of the time will cause two consecutive mispredictions at the start of each loop iteration (the not-taken exit at the end flips the bit, then the first iteration immediately mispredicts). The two-bit saturating counter adds hysteresis: one misprediction alone is not enough to change the prediction.

The counter has four states: 00 (strongly not taken), 01 (weakly not taken), 10 (weakly taken), 11 (strongly taken). The MSB determines the prediction: states 00 and 01 predict not taken; states 10 and 11 predict taken. On each taken outcome, the counter increments toward 11 (saturating there). On each not-taken outcome, it decrements toward 00 (saturating there). This means a branch that is overwhelmingly taken must mispredict twice before its bias changes, filtering out one-off noise.

A practical concern is aliasing: if the BHT has fewer entries than the number of distinct branch PCs in the program, multiple branches will map to the same counter and corrupt each other’s history. Larger tables reduce this interference.

5.11 Branch target buffer (BTB)

Knowing the branch direction (taken vs not taken) is only half the problem. Even if you predict correctly that a branch is taken, you still cannot redirect the fetch to the target address until you know what that address is—and computing the target requires the branch instruction itself to reach decode (to extract the offset) or execute (to add it to the PC). By then, one or more wrong-path instructions have already been fetched.

The Branch Target Buffer (BTB) solves this by caching the last-known target address of each branch, indexed by the branch’s own PC. The slides describe the BTB logic as follows:

  1. The BTB stores the taken-path successor address for each branch it has seen.
  2. It is indexed by the fetch PC, so the lookup happens in parallel with instruction memory access—before the fetched bytes are even decoded.
  3. If the BTB has an entry for the current fetch PC and the direction predictor says “taken,” the next PC is taken directly from the BTB rather than from PC+4. This allows the processor to redirect the fetch immediately, with zero branch penalty when both the direction and target predictions are correct.
  4. On a misprediction (wrong direction or wrong target), the incorrectly fetched instructions are killed, fetch restarts at the correct address, and the BTB entry is updated.

The exam skill here is to narrate mis-speculation recovery: identify which pipeline registers must be flushed, and explain how the PC gets corrected on the cycle the misprediction is discovered.

5.12 “Putting it all together” — two prediction subproblems

Lecture 7 frames modern branch prediction as two independent problems that must both be solved to achieve zero branch penalty: predicting the direction (taken or not taken) and predicting the target address (where to fetch next if taken). The BTB from §5.11 addresses the target problem; this section focuses on the direction problem.

Direction: bi-modal (per-branch saturating counters) is the simplest scheme—one 2-bit counter per BHT entry as described in §5.10. It works well for branches that are overwhelmingly biased in one direction.

Direction: local prediction goes further by tracking the last n outcomes of each specific branch in a local history register. This history bit pattern is then used to index a table of 2ⁿ saturating counters. This scheme excels when a branch has a repetitive, regular pattern—for example, a loop body that takes the branch three times then falls through (pattern 1110 repeated).

Direction: global prediction instead keeps a single global history register (GHR) that records the most recent outcomes across all branches. This is useful when the outcome of one branch is correlated with the outcome of an earlier, different branch. The slides illustrate this with a nested loop:

for (i = 0; i < 100; i++) for (j = 0; j < 3; j++) { /* ... */ }

The outcome of the inner-loop branch (j < 3) is correlated with the outer-loop branch (i < 100)—once the inner loop exits, the outer loop’s history helps predict what comes next. The slides also present a three-branch (B1, B2, B3) pathology where B3’s outcome depends on both B1 and B2 not being taken—a case where local history on B3 alone cannot capture the pattern, but global history can. Named global schemes include gselect (concatenate history with PC bits) and gshare (XOR history with PC bits to reduce aliasing between branches).

Tournament (hybrid) predictors use a meta-predictor to choose between local and global components on a per-branch basis. The meta-predictor itself is a table of saturating counters that learns, over time, which component makes more accurate predictions for each branch. This approach captures the best of both schemes without committing to either.

6. Part E — Memory hierarchy (Lecture 8)

Textbook alignment: H&P (memory hierarchy) Chapter 5, “Large and Fast: Exploiting Memory Hierarchy.”

6.1 Processor–memory gap and software vs hardware

The opening charts in Lecture 8 plot CPU speed and DRAM latency from roughly 1980 to 2010. The two curves diverge dramatically: processor performance improved at roughly 60% per year for much of that period, while DRAM latency improved at only around 7% per year. The result is a processor–memory performance gap—a fast modern CPU stalled behind a single flat DRAM would spend most of its time simply waiting for memory. The entire memory hierarchy exists to bridge this gap by hiding latency and sustaining the bandwidth the CPU needs.

From the perspective of software above the operating system, memory appears as a single large, flat address space—unlimited and uniformly accessible. The hardware reality is very different: at any given moment, a program’s memory traffic is highly concentrated on a small fraction of its address space. This is the principle of locality of reference. Temporal locality means that a recently accessed address is likely to be accessed again soon (due to loops, reused variables, and function calls). Spatial locality means that addresses near a recently accessed address are also likely to be accessed soon (due to sequential code, array traversals, and struct field accesses). Both forms of locality make caching effective.

The hardware design principle that underpins the hierarchy is simple: smaller memories are faster (and more expensive per bit). Systems therefore organize storage as a pyramid—small, fast, expensive structures close to the CPU core, and large, slow, cheap structures further away. A typical server-class system has registers → L1 cache → L2 cache → DRAM → disk. A mobile system might add an L3 cache and use flash in place of a spinning disk. The orders-of-magnitude differences in access time and capacity across these levels are illustrated in the slides: registers hold on the order of 1000 bytes at sub-nanosecond latency; L1 holds tens of kilobytes at about 1 ns; DRAM holds gigabytes at tens of nanoseconds; disk holds terabytes at milliseconds.

The operational model is that the full dataset lives at the lowest (slowest) level. When the CPU accesses data, the hardware copies recently used—and nearby—data up through the hierarchy toward the CPU. On a miss at any level, the hardware fetches the needed block from the next level down, filling the cache and making it available for future accesses.

6.2 Block, hit, miss, rates

The unit of transfer between adjacent levels of the hierarchy is called a block (or cache line). When the CPU accesses a memory address, the hardware checks whether that address’s block is already present in the cache. If it is, that is a hit; execution continues without delay. If it is not, that is a miss, and the hardware must fetch the entire block from the next level down before the access can complete.

The hit rate (or hit ratio) is the fraction of accesses that find their data already in cache. The miss rate is the complement: miss rate = 1 − hit rate. The miss penalty is the additional time (in cycles) needed to service a miss by fetching the block from the next level. Together, these three quantities feed the AMAT formula in §6.8.

6.3 Direct-mapped cache

In a direct-mapped cache, each block in memory maps to exactly one cache line, determined by the formula:

[ \text{line index} = (\text{block address}) \bmod (\text{# lines}) ]

Because many different block addresses can map to the same cache line, the cache must store a tag—the high-order bits of the block address—to distinguish which block currently occupies that line. A valid bit indicates whether the line holds meaningful data at all (it starts invalid on power-up or after a flush).

To perform a cache lookup, the hardware splits the address into three fields (from high to low): tag | index | block offset. The index selects which cache line to inspect. The hardware then checks whether the valid bit is set and whether the stored tag matches the incoming address’s tag. If both conditions hold, it is a hit and the word at the appropriate offset is returned. Otherwise it is a miss: the block is fetched from the next memory level, the line is updated with the new data and tag, and the valid bit is set.

2 26 11010 010 Miss Line 010 gets tag 11, data from Mem[11010]
3 22 10110 110 Hit Same line/tag as step 1
4 26 11010 010 Hit

6.4 Block size tradeoffs

Choosing the right block size involves competing pressures. Larger blocks exploit spatial locality more aggressively: if the program accesses sequential addresses (array traversals, sequential instruction fetch), fetching a bigger chunk on each miss means more future accesses will hit. However, for a fixed total cache capacity, larger blocks mean fewer total lines, which increases conflict misses as more addresses compete for fewer slots. Larger blocks also raise the miss penalty—each miss requires transferring more data from the next level, which takes longer.

Two microarchitectural techniques can hide some of this penalty. Critical word first means the specific word the CPU is waiting for is transferred first across the bus, allowing execution to resume before the rest of the block has arrived. Early restart generalizes this: the processor restarts execution as soon as its needed word arrives in the cache, even if the surrounding words in the same line are still being transferred.

6.5 Misses and pipeline

Cache misses interact directly with the pipeline. On a hit, the cache responds in one cycle and the pipeline proceeds normally. On a miss, the pipeline must stall—it cannot proceed until the requested block has been fetched from the next memory level.

An instruction cache miss stalls the IF stage: the processor cannot fetch the next instruction until the missing instruction’s block arrives, at which point the fetch is replayed. A data cache miss stalls the MEM stage: the load or store must wait until the block is available before it can complete. In both cases, the stall adds extra cycles to the instruction’s execution and inflates the CPI above the ideal value. The exact formula relating miss rate, miss penalty, and CPI impact is given in §6.8.

6.6 Associativity and replacement

Every cache design must answer four questions: (1) where can a block be placed? (2) which block is evicted when a set is full? (3) what happens on a write hit? (4) what happens on a write miss? This section addresses the first two.

A fully associative cache places no restriction on where a block can go—any block address can map to any cache line. This eliminates conflict misses entirely, but requires comparing the incoming tag against every tag in the cache simultaneously, which is expensive in hardware area and power.

An n-way set-associative cache is the practical middle ground. The cache is divided into sets, with each set containing n lines (ways). A block address maps to a specific set using (block address) mod (number of sets), but within that set it can occupy any of the n ways. Only n tag comparisons are needed per lookup. Direct-mapped (1-way) and fully associative are the two extremes of this spectrum.

When a set is full and a new block must be brought in, one existing line must be evicted. Common replacement policies are LRU (least recently used—evict the way that was accessed furthest in the past) and random (simpler hardware, surprisingly competitive in practice for higher associativity). Write policies are covered in §6.7.

6.7 Write policies

When a store instruction hits in the cache, there are two choices for what to do with the written data.

Write-through updates both the cache and the next memory level on every store hit. This keeps the two levels always in sync, simplifying coherence, but can severely inflate CPI if stores are frequent. The slides give a concrete example: if CPI (ignoring memory) is 1, stores are 10% of instructions, and each write to memory takes 100 cycles, the effective CPI becomes 1 + 0.1 × 100 = 11. A write buffer between the cache and memory can absorb the write traffic and let the CPU continue without waiting (as long as the buffer does not fill), significantly reducing this penalty in practice.

Write-back updates only the cache line on a store hit, marking it dirty. The modified data is only written back to the next level when the dirty line is eventually evicted. This dramatically reduces memory write bandwidth for workloads that write to the same location repeatedly, since only the final value is sent downstream. The cost is complexity: the cache must track dirty bits and handle writeback during eviction.

When a store misses the cache, there is also a choice. Write-allocate (typical with write-back) fetches the missed block into the cache first, then updates it—betting that the same location will be written or read again soon. No-write-allocate (also called write-around, typical with write-through) writes the data directly to the next level without bringing the block into cache, avoiding cache pollution for cold writes that are unlikely to be reused.

6.9 Three Cs and unified vs split (numeric example on slides)

Cache misses fall into three categories, commonly called the Three Cs. Compulsory misses (cold misses) happen the very first time a block is accessed—the block has never been in the cache before. These are unavoidable and represent a lower bound on misses. Capacity misses occur because the cache is not large enough to hold the program’s entire working set; even a fully associative cache of the same size would still suffer these misses. Conflict misses are unique to caches with limited associativity: two blocks that map to the same set repeatedly evict each other even though the cache as a whole has free space elsewhere. Increasing associativity eliminates conflict misses at the cost of hardware complexity.

The slides include a numeric example comparing a 32 KB unified cache against a split 16 KB instruction cache + 16 KB data cache with a 50-cycle miss penalty and 1-cycle hit time. In the unified case, data accesses (loads/stores) pay an extra cycle because they must share the single cache port with instruction fetches. Across a workload with 75% instruction fetches and 25% data accesses:

  • Unified 32 KB: miss rate 1.99%, data hit costs 2 cycles (shared bus overhead). AMAT = 0.75 × (1 + 1.99% × 50) + 0.25 × (2 + 1.99% × 50) = 2.24 cycles.
  • Split 16+16 KB: I$ miss rate 0.64%, D$ miss rate 6.47%, no sharing penalty. AMAT = 0.75 × (1 + 0.64% × 50) + 0.25 × (1 + 6.47% × 50) = 2.05 cycles.

Even though the split cache has higher weighted miss rate (2.10% vs 1.99%), it wins on AMAT because eliminating the shared-port +1 cycle overhead for data hits outweighs the higher D$ miss rate. The general levers for reducing AMAT are: increase block size (up to a point), increase cache capacity, increase associativity, add more cache levels, control inclusion between levels, and use read-before-write buffering (§6.11).

6.10 Multilevel: effective L1 miss penalty to memory

When a second-level cache (L2) sits between L1 and main memory, the effective penalty for an L1 miss is no longer simply the time to go to DRAM—it is the time to either hit in L2 or miss all the way to DRAM. The slides work through the following example with main memory at 200 cycles:

  • 1-way (direct-mapped) L2: L2 hit time = 10 cycles, local L2 miss rate = 25%. Effective L1 miss penalty = 10 + 0.25 × 200 = 60 cycles.
  • 2-way set-associative L2: L2 hit time = 11 cycles, local L2 miss rate = 20%. Effective L1 miss penalty = 11 + 0.20 × 200 = 51 cycles.

Even though the 2-way L2 takes 1 extra cycle on a hit, its lower miss rate more than compensates—the end-to-end L1 miss cost drops from 60 to 51 cycles. This illustrates a general principle: the miss rate reduction from higher associativity can outweigh the hit time increase, especially when the miss penalty to the next level is large.

The slides also raise the inclusion property: should every line present in L1 also be present in L2? Inclusion simplifies coherence (an L1 eviction always finds the line in L2) but wastes L2 space. Non-inclusive designs must handle the case where an L1 eviction is not in L2, requiring a write-back to L2 or DRAM.

6.11 Write buffer and read–write ordering (slide 41)

A write buffer sits between the cache and the next memory level. When a write-through cache needs to write data downstream, instead of stalling the CPU until the write completes, the data is placed in the write buffer and the CPU continues immediately. The write buffer drains to memory in the background. This decouples CPU speed from memory write bandwidth as long as the buffer does not fill.

A subtlety arises with read-write ordering. Suppose a store to address M[512] has been placed in the write buffer but not yet committed to memory. If a load to a different address (M[1024]) misses the cache and the hardware services that miss by reading from memory, it might then issue a load to M[512] while the store to M[512] is still only in the write buffer. If the cache line for M[512] is not in the cache, the load could read the old, stale value from memory rather than the newer value in the write buffer. The fix requires either snooping (checking the write buffer on every load miss before going to memory) or stalling load misses until the write buffer drains to memory. The slides name the read-before-write technique in the context of miss penalty reduction, noting that this is not a full memory consistency model—just a correctness constraint on the local write buffer.

6.13 VIPT (virtual index, physical tag)

A Virtually Indexed, Physically Tagged (VIPT) cache achieves a useful speed trick. Address translation (virtual → physical) normally requires a TLB lookup, which adds latency to every cache access. However, the low-order bits of a virtual address below the page offset are identical in the virtual and physical addresses (they are not translated at all). VIPT exploits this: the low bits are used to index into the cache in parallel with the TLB lookup. By the time the TLB returns the physical page number (which forms the tag), the cache has already selected the right set. The physical tag comparison then determines whether it is a hit.

The constraint on VIPT is that the number of index bits must not exceed the number of page-offset bits. If the index reaches into the page number bits, then two different virtual addresses that translate to the same physical address could index into different cache sets—creating alias (synonym) bugs where the same physical data appears in multiple places in the cache. This constraint is sometimes expressed as: the product of associativity × the number of sets must not exceed the page size. Page coloring is an OS technique for managing aliasing when a design violates this constraint.

The slides also note the page size tradeoff: larger pages mean smaller page tables, more efficient bulk I/O transfers, fewer TLB misses (since each TLB entry covers more memory), and potentially larger VIPT caches. Smaller pages cause less internal fragmentation (wasted space within a page when the allocation does not fill it exactly).

6.14 Near-memory and processing-in-memory (PIM)

As DRAM latency has not kept pace with CPU speed (the gap shown in §6.1), a growing fraction of execution time and energy is consumed simply moving data between the processor and memory. Processing-in-Memory (PIM) proposes to reduce this data movement by placing compute capability close to or inside the memory itself.

The lecture deck includes an excerpt from Loh et al. (WoND 2015), which surveys the PIM design space. The paper notes that PIM is not a new idea but has become more practical with 3D die stacking, which allows logic and DRAM layers to be integrated vertically with very high-bandwidth connections between them.

The paper categorizes PIM designs along a spectrum of software visibility. Non-compute in-memory operations (ECC correction, self-test (BIST), prefetching) are transparent to software. Software-visible PIM falls into three tiers: BPO (bounded programmable operations)—fixed, hardware-accelerated operations like vector reduce; CPO (compound programmable operations)—library-level operations like scatter/gather; and fully programmable PIM (GPU-style cores near memory). The paper’s Figure 2 contrasts these tiers on a loop–reduce–library–PIM_fork axis, arguing that the middle fixed-function tier often gives the best efficiency per silicon area. F.1 Shifts, multiplication, representation (Background A.1, Discussion IV)

Left shift and powers of two: For integers, a left shift by (k) bits (vacated low bits filled with zeros) multiplies the unsigned interpretation by (2^k), modulo (2^n) in an (n)-bit word. In two’s complement, the same bit pattern operation is often distinguished as logical vs arithmetic shift; for non-negative values that stay in range (no overflow into the sign bit), one left shift multiplies by exactly two per shift. A single shift implements multiply only by one power of two (2^k) (choose (k) as the shift amount); it does not implement multiply by arbitrary constants (e.g. 3, 7, 10, 33). Examiners expect separation of: (1) algorithmic limitation: one shift ≠ multiply by a non-power-of-two; (2) representation / overflow: even for powers of two, signed overflow can make a shifted result differ from the mathematical product in a fixed width; in C, left shift of negative or overflow cases can be undefined — in hardware/architecture the usual point remains: no integer (k) makes ((x \ll k) = 3x) for all (x) in range.

F.2 Problem 1(a) — “Left shift is enough for multiply”

C.2 — Consumers are M- or MR-format: Register-number logic starts like R-type consumer in EX — two operand values for EX after memory reads. Address / memory RAW: dependencies are not only Rd_producer = Rs_consumer but producer wrote memory at A and consumer reads A; two different register names can hold the same numeric address — true dependency detection may require comparing register contents (address equality), not only indices.

C.3 — Producer is M-format (writes only memory, not a register file dest): Consumers: (1) Another M- or MR instruction whose source reads that address in EX; (2) I-format load that loads that address in MEM. Timing (handout): M-format value to store ready end of EX; commits in MEM. EX→EX forwarding can bypass stale memory for M/MR consumers; address equality on appropriate register contents, analogous to R-type EX→EX but with addr match. After producer finishes MEM, reading memory is correct; MEM→EX is an optimization, not always required for correctness for M→M/MR. For load consumer, address known after EX; forwarding may need MEM→MEM; after producer MEM, plain load is correct; forwarding = performance.

C.4 — Producer is MR-format: Like R-type for register Rd — result end of EX; on EX/MEM, MEM/WB pipeline registers. From EX/MEM (value just finished EX), if EX/MEM.RegRd ≠ $0: Forward to ID (branch operands) if EX/MEM.Rd equals IF/ID.Rs or IF/ID.Rt. Forward to EX if EX/MEM.Rd equals ID/EX.Rs or ID/EX.Rt. From MEM/WB, if MEM/WB.RegRd ≠ $0 and EX→EX conditions are false (priority to younger result): Forward to EX if MEM/WB.Rd equals EX/MEM.Rs or EX/MEM.Rt (handout: match pipeline diagram naming for the instruction in MEM`). Distance: up to distance 2** consumers may still need bypass for MR (same spirit as classic MIPS).

Walkthrough (M/MR handout — using §F.6 answers)

Recognize the format and hazard type

  1. Read the instruction as M or MR: are rs/rt/rd data or pointers? M = write memory at Rd’s address; MR = write a data Rd in the register file (MEM “bogus” for that result).
  2. Register-number RAW: if old and new use the same Rd/Rs/Rt indices as classic MIPS, apply the same EX/MEM vs MEM/WB rules as in §4–5.
  3. Address-based hazard: if M wrote *a and a later op reads *a, check whether register numbers differ but pointers are equal (same 0x… in $t0 and $t1)—then you need value compare of address contents, not just R# match.

M-format producer → M/MR or load consumer**

  1. M commits the store in MEM; value is ready after EX (for the data) and after MEM (for memory).
  2. A consumer in EX that loads the same logical address: either stale if you only snoop regs, or use bypass from M in EX/MEM with addr match, or wait for mem to be correct. MEM→MEM is optional optimization (§5.6).

MR producer (register Rd)

  1. Treat like R-type for Rd: result after EX; forward EX/MEM and MEM/WB to ID/EX with RegWrite and $Rd \neq 0$, and revised younger-producer priority as in §4.6.
  2. Distance 1 and 2: same spirit as standard RAW chains.

Extra practice (M/MR, handout): P6 — M-format writes sum of *p and *q to *r; next MR reads *r and *s into a register. State stalls and forwarding paths with and without address equality. P7 — Counterexample where register-number-only forwarding fails but address-content forwarding succeeds — use $t0 and $t1 with different register numbers but both hold pointer value 0x1000 (same address).

F.7 Quick reference card (discussion)

Shift-only: powers of two; counterexample ×3. Add-shift: popcount of chosen multiplier. TIMS word-addressed, dest in mem → long code. Matrix: not IF/WB dest for forwards as in key; no EX→MEM. Branch in ID → EX/MEM→ID. M-format: memory RAW; addr content. MR: R-like Rd forward.

9.1 Single-cycle MIPS: split instruction and data memories (Harvard-style)

Idea: One clock cycle must cover the slowest path (here, lw): I-fetch, reg read, ALU (address), D-read, reg write. I-memory and D-memory are separate so fetch and load/store are not one structural port in the same cycle.

flowchart TB
  subgraph scclk [One clock cycle - edge to same edge]
    subgraph ifetch [IF]
      PC[PC register] -->|address| IMem[Instruction memory]
      IMem -->|32-bit instruction| IR[Instruction word]
    end
    subgraph idecode [ID]
      IR --> Dec[Control decode - opcode, funct]
      Dec --> Ctl[Control unit]
      IR -->|rs,rt| RFR[Regfile read ports]
    end
    subgraph iexe [EX]
      RFR -->|rs data| AluA[ALU A]
      MuxB[Mux ALUSrc] -->|B| AluA
      AluA --> AluR[ALU result]
    end
    subgraph imem [MEM]
      AluR -->|load-store addr| DMem[Data memory]
      RFR -->|store data| DMem
    end
    subgraph iwb [WB]
      MuxM[Mux MemtoReg] -->|write data| RFW[Regfile write port]
      Ctl --> RFW
    end
  end
  Ctl -.->|RegWrite, MemtoReg, ...| MuxM
  Ctl -.->|ALUSrc| MuxB
  DMem -->|load data| MuxM

9.2 PC next-value logic: sequential, branch, jump

Idea: PC+4 is default; if beq and Zero, take branch target; if j/jal, take Jump target. Priority is typically Jump over Branch in combined mux chains (one instruction at a time).

flowchart TB
  PC0[PC] --> Pc4[PC+4 adder]
  Pc4 -->|input 0| MxBr[Multiplexer: sequential vs branch]
  Pc4 -->|base| BrTgt[Branch target add: PC+4 + beq offset, shifted 2]
  BrTgt -->|input 1| MxBr
  SelBr[Select = Branch AND Zero] -.->|control| MxBr
  MxBr -->|pre-jump| MxJ[Multiplexer: if Jump then jump, else MxBr]
  Pc4 -.->|high bits| Jc[Jump target: 4 bits of PC+4, 26-bit word, 2 bits 0]
  Jc -->|jump target| MxJ
  SelJ[Select = Jump] -.->|control| MxJ
  MxJ --> PCN[PC next]
  PCN --> PC0

Separately, ALU does R[rs]−R[rt] and Zero for equality on beq; SelBr comes from that and from main control (Branch=1 for beq). SelJ is one for j/jal only.

9.3 Five-stage pipeline: time-multiplexed path with pipeline registers

Idea: Same functional resources as the single-cycle design, but latches (IF/ID, ID/EX, EX/MEM, MEM/WB) cut timing into five shorter cycles. T_pipe ≈ max(stage delay); throughput approaches 1 IPC when the pipe is full and hazards are absent.

  L2 --> sEX[ALU, branch math if in EX in some designs - course uses branch in ID]
  sEX --> L3[EX/MEM latch]
  L3 --> sMEM[D-memory]
  sMEM --> L4[MEM/WB latch]
  L4 --> sWB[Writeback mux, regfile write]
  sWB --> RegWb[Regfile - write in WB]
  RegWb -.->|architectural state| sID

9.4 Data forwarding (bypass) to the EX stage

Idea: For RAW on adjacent instructions, the consumer’s ALU inputs are muxed from EX/MEM (result just finished EX) or MEM/WB (value after MEM for load) instead of the stale regfile read in ID/EX.

Priority when both match the same Rd: use EX/MEM (younger producer) over MEM/WB (Lect06 revised rule).

flowchart TB
  RsD[Reg read from ID/EX: Rs]
  RtD[Reg read from ID/EX: Rt]
  M1A[Mux: pick Rs from RF vs EX-MEM vs MEM-WB - priority: EX if match]
  M1B[Mux: pick Rt same way]
  RsD --> M1A
  RtD --> M1B

9.5 Load–use hazard: detection and 1-cycle stall

Idea: Load data exists after MEM; the next instruction cannot get it at EX in the same schedule without a bubble. Hazard when ID/EX.MemRead and ID/EX.Rt equals IF/ID.Rs or IF/ID.Rt. Stall PC and IF/ID; insert nop in ID/EX.

  AndH -->|no| Nop2[Normal advance]
  IDEXL[ID/EX load] --> HDU
  IFID[IF/ID consumer] --> HDU

9.6 Control hazard: branch in ID, fall-through, then flush

Idea: IF may have fetched the not-taken path; if taken, instructions after the branch in the fetch/decode pipe are wrong path and must be flushed (squash IF/ID or replace with nops) once the branch resolves in ID.

9.7 Memory hierarchy: logical pyramid and traffic on miss

Idea: Smaller, faster levels hold working sets copied from larger, slower levels. On miss, a block (line) is moved up; off-chip (DRAM) and storage are last.

9.8 Direct-mapped cache: index, tag, valid, and compare

Idea: Four states; MSB often encodes predict T vs NT; misprediction nudges the counter, hysteresis against noise.

9.12 Branch target buffer (BTB) in fetch path (conceptual)

9.14 Multilevel cache: L1 miss penalty as seen by the CPU (L2 example)

Idea: When L1 misses, the cost is L2 hit time plus L2’s local miss rate times the cost to the next level (e.g. main memory), as in the 1-way vs 2-way L2 design comparison on the course slides.

Single-cycle datapath and control (Lect04)

  • Lifecycle: Fetch → Decode/operands → Execute/address → Memory (if load/store) → Writeback / PC update.
  • Field layouts: R-type (op, rs, rt, rd, shamt, funct); I-type (op, rs, rt, imm16); J-type (op, addr26).
  • Branch target: (PC+4) + sign_ext(offset16) << 2.   Jump target: PC+4[31:28] ‖ addr26 ‖ 00.
  • ALU control: from ALUOp (2 bits, coarse class) + funct field (R-type only).
  • Main control signals: RegDst, RegWrite, ALUSrc, MemRead, MemWrite, MemtoReg, Branch, ALUOp, Jump — set by opcode.
  • Critical path: load instruction — I-mem → regfile read → ALU → D-mem → regfile write. This sets T_clk for the whole design.
  • Hardware blocks: mux, ALU, D flip-flops, clocking discipline; split I-mem / D-mem.

Pipelining (Lect05–Lect07)

TopicKey points
CPI & throughput Clock period set by slowest stage; ideal speedup ≈ number of stages when stages balance and pipe stays full. Throughput ↑ but single-instruction latency does not decrease.
ISA advantages MIPS: fixed 32-bit instructions, few formats, load/store architecture, aligned operands.
Structural hazard Shared single-ported memory (IF vs MEM). Fix: separate I-cache and D-cache.
Data hazard (RAW) Forwarding from EX/MEM and MEM/WB. Need RegWrite and Rd ≠ $0. Double hazard: prefer EX/MEM over MEM/WB for same Rd (revised Lect06 rule).
Load-use stall Load data not available until after MEM; 1-cycle stall + hazard detection in ID. PC and IF/ID frozen; ID/EX squashed to bubble. Fix with code scheduling (place independent work between load and consumer).
Control hazard Branch in ID: wrong-path instructions may enter IF; flush on taken. Forwarding to ID comparator for branch operands. Stall counts: R-type at dist 1 → 1 stall; load at dist 1 → 2 stalls; dist 2+ → 0 stalls (with forwarding).
Branch prediction Static: backward taken / forward not-taken heuristic. Dynamic: 2-bit saturating counter FSM. BTB: indexed by fetch PC, supplies predicted next PC; flush and restart on mispredict.

Memory hierarchy (Lect08)

TopicKey points
Locality & stack Temporal & spatial locality; hierarchy: registers → L1 → L2 → DRAM → storage.
Basics Block (line) = transfer unit. Hit rate, miss rate = 1 − hit rate, miss penalty = cycles to fetch block from next level.
Direct-mapped Index = (block addr) mod (#lines). Address = Tag | Index | Block offset. Valid bit; tag compare on every access.
Associativity n-way set-associative: set = (block#) mod (#sets); n parallel tag compares. Replacement: LRU, random. Fully-assoc → no conflict misses.
Block size Larger → better spatial locality but higher miss penalty and possible conflict increase.
Write policies On hit: write-through (cache + memory) vs write-back (dirty bit, defer write). On miss: write-allocate vs no-write-allocate.
Performance Exec time = (CPU cycles + mem stall cycles) × T. AMAT = hit time + miss rate × miss penalty. 3Cs: compulsory, capacity, conflict.
Multilevel Effective L1 miss penalty = L2 hit time + L2_local_miss_rate × L2_miss_penalty. Split I$/D$ vs unified: use weighted AMAT formula.
Virtual memory & TLB VPN + offset → TLB → PFN + offset = PA. Page fault: OS handles. VIPT: index uses VA page-offset bits; tag compared in PA space.
Near-memory (PIM) Motivation: memory bandwidth/latency/energy gap. Taxonomy: BPO, CPO, programmable (awareness-level).

Part (b.i) — Doug's claim: shift + add suffices
Demonstrate how multiply can be implemented using only left shift and add (positive numbers).
Core insight: Every integer is a sum of powers of 2 (binary representation).

Example: 13 = 1101₂ = 8 + 4 + 1 = 2³ + 2² + 2⁰

So: x × 13 = (x << 3) + (x << 2) + (x << 0)

Each term is a left shift; we accumulate with adds.
This is binary long multiplication — the same algorithm as decimal long multiplication, but in base 2.
Add-and-shift algorithm (multiplicand M, multiplier Q, accumulator A = 0):

For each bit of Q from LSB to MSB:
  1. If current bit = 1:  A ← A + M
  2. Always:  M ← M << 1  (double M for next place value)
After all bits: A = M_original × Q.
Adds used = popcount(Q)  ·  Shifts used = bit-width of Q.
Why it's correct:

At iteration i, M = M_original × 2i. When bit i of Q is 1, we add M_original × 2i to A.

Final A = M_original × Σ (Qi × 2i) = M_original × Q ✓
Binary representation guarantees every integer is a sum of powers of 2. Shifts handle the powers; adds handle the sum.

Part (b) — Simultaneous EX→EX and MEM→EX?
Can EX-to-EX and MEM-to-EX forwarding fire at the same time? If so, when?
The setup: An instruction entering EX has two source registers. Each ALU input has its own independent forwarding mux that can select from:

 • Register file (no hazard)
 • EX/MEM latch (EX-to-EX path)
 • MEM/WB latch (MEM-to-EX path)

Could input A and input B each need a different bypass path simultaneously?
The two muxes are independent — there's no hardware reason they can't both be active at once.

• I2 is in MEM → $s1 in EX/MEM latch → EX-to-EX for input B
• I1 is in WB → $s0 in MEM/WB latch → MEM-to-EX for input A

M-format & MR-format — Forwarding Conditions

▼
Background — The three formats
R-format, M-format, and MR-format share opcodes but differ in what register values mean and where results go.
R-format (baseline MIPS):

• Rs, Rt → register data values (read at ID)
• ALU result → written to Rd register at WB
• MEM stage: unused (bogus)
• Forwarding: EX/MEM and MEM/WB → EX; EX/MEM → ID for branches
M-format:

• Rs, Rt → register values are memory addresses
• Source data fetched from memory at those addresses — within EX
• Result stored to memory at address in Rd — at MEM (like a store)
• No register write — produces a memory update, not a register update
Think: "M for memory destination." Sources and destination are all in memory; registers just hold pointers.
MR-format:

• Sources: same as M-format — Rs, Rt hold memory addresses; data fetched in EX
• Result: written to register Rd as data — at WB (like R-format)
• MEM stage: bogus (no memory write)
• As a producer, behaves identically to R-format
Think: "MR = Memory sources, Register destination." Reads from memory, but stores result in a register.
Part (a) — Consumer is M- or MR-format
How should forwarding work when the consuming instruction is M- or MR-format?
First layer — like R-format:

M- and MR-format both consume two register values at EX, just like R-format. If a prior instruction wrote to register $t0 and the M/MR consumer reads $t0, that's a normal register-number RAW.

Apply EX-to-EX (EX/MEM.Rd == ID/EX.Rs or .Rt) and MEM-to-EX (MEM/WB.Rd == ID/EX.Rs or .Rt) forwarding conditions exactly as in standard MIPS.
Second layer — the address complication:

In M/MR, register values are addresses. Two different registers can hold the same address: 
add $t0, ..., ...     # $t0 = 0x1000
add $t1, ..., ...     # $t1 = 0x1000 (same addr!)
M-inst Rd, $t0, $t1   # reads mem[$t0] and mem[$t1]
If an earlier M-format instruction wrote to address 0x1000 (via a different register name), a register-number check alone would miss this dependency.
True dependency detection may need to compare register contents (the actual address values), not just register indices.
Two forwarding layers:

1. Register-number check: Standard EX-to-EX and MEM-to-EX. Covers cases where the same register holds a pointer that was recently written.

2. Address-content check: Compare the address value in the producer's Rd against the address values in the consumer's Rs/Rt. Catches dependencies between different register names that hold the same address.
Part (b) — Producer is M-format
How should forwarding work when an M-format instruction is the producer?
Key fact: M-format writes to memory, not a register.

The result is computed at end of EX and sits in EX/MEM. It's committed to memory address A (in Rd) during MEM.

Any consumer that reads from address A is dependent. Detecting this requires comparing address values, not register numbers, because A is a memory location — not tracked by register renaming.
Consumer is M- or MR-format (needs data at EX):

M producer's result is ready at end of EX → in EX/MEM latch. M/MR consumer needs operand data at the start of EX (for its own memory reads).

EX-to-EX forwarding with address-content matching:
Check: value in producer's Rd register == value in consumer's Rs or Rt register (both are memory addresses — compare the pointer values).
MEM-to-EX is less critical: once the producer finishes MEM, consumers can safely read directly from memory. But MEM-to-EX saves a memory access and hides any latency.
Consumer is a load (needs data at MEM):

A load reads from memory in MEM. The M-format producer writes at MEM.

MEM-to-MEM forwarding (optional performance optimization):
• The load's effective address isn't known until its own EX completes
• By the time the load enters MEM, the producer has already written → load could just read normally
• MEM-to-MEM forwarding can skip the memory read, saving a cache access
Not required for correctness — but useful if the access might miss in cache, or to reduce memory port pressure.

Part (c) — Producer is MR-format
How should forwarding work when an MR-format instruction is the producer?
MR-format producers are equivalent to R-format producers:

MR writes its result to register Rd at WB — same as R-format. Result is ready at end of EX and sits in the EX/MEM latch.

Forwarding needed for consumers at up to distance 2. Consumers at distance ≥3 read the register file normally.
Use register-number matching (not address-content), because MR writes a data value to a register — no memory addressing involved on the destination side.
Forward from EX/MEM (producer just exited EX):

Condition: EX/MEM.RegWrite = 1 AND EX/MEM.Rd ≠ $zero

→ ID (branch consumer):
 EX/MEM.Rd == IF/ID.Rs  OR  EX/MEM.Rd == IF/ID.Rt

→ EX (R, M, MR, or I-format consumer):
 EX/MEM.Rd == ID/EX.Rs  OR  EX/MEM.Rd == ID/EX.Rt
Forward from MEM/WB (producer in MEM, consumer at distance 2):

Condition: MEM/WB.RegWrite = 1 AND MEM/WB.Rd ≠ $zero
AND EX-to-EX conditions are NOT satisfied (priority to the younger — closer — producer)

→ EX:
 MEM/WB.Rd == EX/MEM.Rs  OR  MEM/WB.Rd == EX/MEM.Rt
MR-format producers: identical forwarding conditions to R-format. Register-number matching. Forward from EX/MEM and MEM/WB to EX (and EX/MEM to ID for branches).
