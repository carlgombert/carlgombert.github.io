Application Layer

The application layer is where network applications and their application-layer protocols reside. It is the "top" layer of the Internet stack, providing services directly to the user.

11. Principles of Network Applications

Developing network applications involves writing programs that run on different end systems and communicate with each other over the network. Crucially, application software does not need to be written for network-core devices (routers or switches), as they function at lower layers.

11.1 Network Application Architectures

There are three primary paradigms for structuring network applications:

  1. Client-Server Architecture:

    • Server:
      • Always-on host.
      • Permanent IP address.
      • Often resides in data centers for scaling.
    • Clients:
      • Communicate with the server.
      • May be intermittently connected.
      • May have dynamic IP addresses.
      • Do not communicate directly with each other.
    • Examples: Web, E-mail, FTP.

  2. Peer-to-Peer (P2P) Architecture:

    • No always-on server.
    • Arbitrary end systems (peers) directly communicate.
    • Self-Scalability: New peers bring new service capacity (upload bandwidth) as well as new service demands.
    • Peers are intermittently connected and change IP addresses.
    • Challenges: Complex management due to the highly decentralized nature.
    • Examples: P2P file sharing (BitTorrent), Bitcoin.

  3. Hybrid Architecture:

    • Combines elements of both. For example, a centralized server might be used to find the address of a remote party (directory service), but the actual data transfer happens peer-to-peer.
    • Examples:
      • Skype/VoIP: Centralized server for finding users; direct client-to-client connection for the call.
      • Instant Messaging: User registers IP with a central server; chatting between users is P2P.

11.2 Processes Communicating

A process is a program running within a host.

  • Within the same host, processes communicate using inter-process communication (defined by the OS).
  • Processes on different hosts communicate by exchanging messages.

Client vs. Server Processes

  • Client Process: The process that initiates the communication.
  • Server Process: The process that waits to be contacted.
  • Note: In P2P applications, a peer can have both a client process and a server process.

Sockets

A process sends/receives messages to/from the network through a software interface called a socket.

  • Analogy: A socket is like a door. The sending process shoves the message out the door, relying on the underlying transport infrastructure to deliver it to the socket at the receiving process.

Addressing Processes

To receive a message, a process must have an identifier, which includes:

  1. IP Address: Identifies the specific host on the network (32-bit for IPv4).
  2. Port Number: Identifies the specific process/service on that host.
    • HTTP: Port 80
    • Mail (SMTP): Port 25

11.3 Application-Layer Protocols

An application-layer protocol defines:

  • Types of messages exchanged (e.g., request, response).
  • Syntax: Fields in the message and how they are delineated.
  • Semantics: Meaning of the information in the fields.
  • Rules: When and how a process sends and responds to messages.

12. Web and HTTP

12.1 HTTP Overview

The HyperText Transfer Protocol (HTTP) is the Web's application-layer protocol. It follows a client/server model:

  • Client: Browser that requests, receives, and displays Web objects.
  • Server: Web server that sends objects in response to requests.

Web Objects and URLs

A Web page consists of objects (HTML files, JPEG images, Java applets, audio files, etc.). Most pages consist of a base HTML file that references several other objects. Each object is addressable by a URL (Uniform Resource Locator): http://www.someschool.edu/someDept/pic.gif

  • Host Name: www.someschool.edu
  • Path Name: /someDept/pic.gif

Transport Layer: TCP

  • HTTP uses TCP as its underlying transport protocol.
  • The client initiates a TCP connection (creates a socket) to the server on port 80.
  • The server accepts the connection.
  • HTTP messages are exchanged.
  • TCP connection is closed.

[!NOTE] HTTP is Stateless The server maintains no information about past client requests. If a client asks for the same object twice, the server sends it again. Stateless protocols are easier to design and manage because the server doesn't need to keep track of complex state variables.

12.2 HTTP Connections

Non-Persistent HTTP

  • At most one object is sent over a single TCP connection.
  • Downloading multiple objects requires multiple TCP connections.
  • Process:
    1. TCP connection opened.
    2. Request/Response exchanged.
    3. TCP connection closed.

Persistent HTTP (HTTP/1.1)

  • The server leaves the connection open after sending a response.
  • Subsequent HTTP messages between the same client/server can be sent over the same connection.
  • Pipelining: The client can send requests as soon as it encounters a referenced object (e.g., while still receiving the base HTML), rather than waiting for each object to finish.

12.3 Performance: RTT and Response Time

Round-Trip Time (RTT)

Definition: The time it takes for a small packet to travel from client to server and back.

Response Time Calculations

  1. One RTT to initiate the TCP connection.
  2. One RTT for the HTTP request and the first few bytes of the HTTP response to return.
  3. File Transmission Time: The time to push all bits of the file into the link.

Total Response Time (Non-Persistent): $$T_{resp} = 2 \times RTT + \text{transmission time}$$

Total Response Time (Persistent):

  • As little as one RTT for all referenced objects (after the initial connection and base HTML RTTs), significantly cutting down latency.

[!IMPORTANT] Exam Examples: Calculating RTTs Assume the client knows the IP address of the server. Transmission time for each object (base HTML or JPEG) is 0.5 RTT.

Scenario 1: Base Page Only

  • TCP Setup: 1 RTT
  • HTTP Request/Response: 1 RTT
  • Transmission: 0.5 RTT
  • Total: 2.5 RTT

Scenario 2: Base Page + 1 JPEG (Non-Persistent HTTP 1.0)

  • Base Page: 2.5 RTT
  • New TCP Setup for JPEG: 1 RTT
  • HTTP Request/Response for JPEG: 1 RTT
  • Transmission for JPEG: 0.5 RTT
  • Total: 2.5 + 2.5 = 5 RTT

Scenario 3: Base Page + 1 JPEG (Persistent HTTP 1.1)

  • Base Page (Setup + Request + Trans): 2.5 RTT
  • Request/Response for JPEG (Same Connection): 1 RTT
  • Transmission for JPEG: 0.5 RTT
  • Total: 2.5 + 1.5 = 4 RTT

Parallel Connections

To mitigate the slowness of non-persistent HTTP, modern browsers often open multiple parallel TCP connections to fetch referenced objects simultaneously.

12.4 HTTP Message Formats

HTTP messages are in human-readable ASCII format.

HTTP Request Message

  • Request Line: Method (GET, POST, HEAD, etc.), URL, and HTTP Version.
  • Header Lines: Metadata like Host, User-Agent, Accept-Language, Connection (e.g., keep-alive).
  • Entity Body: Used in POST and PUT methods to send data to the server.

HTTP Request Methods

Method Description
GET Requests data. Can send user data in the URL (e.g., search?q=monkeys).
POST Sends data in the entity body (e.g., form input, creating a resource).
HEAD Requests only the headers (no body). Used for debugging/testing.
PUT Uploads/replaces a file at the specified URL.
DELETE Deletes a file at the specified URL.

HTTP Response Message

  • Status Line: Protocol version, Status Code, and Status Phrase.
  • Header Lines: Date, Server, Last-Modified, Content-Length, Content-Type.
  • Entity Body: The actual data (e.g., requested HTML file).

12.5 HTTP Status Codes

These appear in the first line of the server response.

  • 200 OK: Request succeeded; object is in the response.
  • 301 Moved Permanently: Object has a new location (specified in Location: header).
  • 400 Bad Request: Server didn't understand the request.
  • 404 Not Found: Document not found on this server.
  • 505 HTTP Version Not Supported.

12.6 Conditional GET

The goal of a conditional GET is to avoid sending an object if the client (or a proxy cache) already has an up-to-date version in its cache. This saves bandwidth and reduces latency.

  • Mechanism:
    1. Cache: The client/proxy stores the object along with the `Last-Modified` date from the server.
    2. Request: In a subsequent request, the client includes an `If-modified-since: ` header.
    3. Server Response:
      • If the object has not changed, the server returns 304 Not Modified with no entity body.
      • If the object has changed, the server returns 200 OK with the new object.

13. Cookies: Maintaining User State

Although HTTP is stateless, many applications (e.g., shopping carts, authentication, personalized content) require the server to identify users across multiple requests.

13.1 Cookie Components

To maintain state, the cookie system uses four components:

  1. Response Header: Set-cookie: header in the HTTP response message.
  2. Request Header: Cookie: header in subsequent HTTP request messages.
  3. Client-Side File: A cookie file kept on the user's host, managed by the browser.
  4. Backend Database: A database at the Web site to associate cookie IDs with user data/actions.

13.2 How Cookies Work

  1. First Visit: Client sends an ordinary HTTP request. Server creates a unique ID and an entry in its database. Server responds with a Set-cookie: ID header.
  2. Subsequent Visits: The browser sees the cookie for that site and includes it in the request: Cookie: ID.
  3. State Tracking: The server uses the ID to look up the user's past actions or preferences in its database.

13.3 Summary of State Management

Aspect Stateless (Pure HTTP) Stateful (with Cookies)
Server Complexity Low (no memory of past) High (must manage database)
Reliability High (survives crashes) Lower (database/session loss issues)
User Privacy Higher Lower (enables tracking/profiling)

14. Network Diagnostic Tools

Several tools are used to interact with or observe HTTP:

  • telnet: Can be used to manually type HTTP requests to a server on port 80.
  • curl / wget: Command-line tools to download objects and view headers.
  • Wireshark: A packet sniffer that captures and displays all network traffic on an interface. It is essential for debugging protocol interactions and viewing the "raw" bytes of HTTP messages.

15. The Domain Name System (DNS)

The DNS is a distributed database implemented in a hierarchy of many name servers and an application-layer protocol that allows hosts to query the database.

15.1 DNS Services

  • Hostname to IP Address Translation: The primary role of DNS.
  • Host Aliasing: A host can have one or more alias names (e.g., a simple name like www.google.com for a complex canonical name).
  • Mail Server Aliasing: Allows mail servers to have simpler aliases.
  • Load Distribution: Replicated Web servers can be associated with a single hostname; DNS rotates the list of IP addresses returned to distribute traffic.

15.2 The DNS Hierarchy

DNS uses a hierarchical structure to provide scalability and resilience.

  1. Root DNS Servers: There are 13 logical root servers worldwide (replicated hundreds of times). They provide the IP addresses of TLD servers.
  2. Top-Level Domain (TLD) Servers: Responsible for domains like .com, .org, .net, .edu, and country codes (.uk, .fr).
  3. Authoritative DNS Servers: An organization's own servers that provide the actual hostname-to-IP mappings for that organization's hosts.
  4. Local DNS Servers: While not strictly part of the hierarchy, every ISP has a "default" local DNS server. When a host makes a query, it goes to the local DNS server first.

15.3 Name Resolution Processes

  • Iterated Query:
    • The contacted server replies with the name of the next server to contact ("I don't know this name, but ask this server").
    • The burden of resolution stays on the local DNS server.
  • Recursive Query:
    • The contacted server takes on the task of finding the mapping itself by querying other servers.
    • This puts a heavy load on upper-level servers (like root servers) and is generally avoided at those levels.

15.3.1 The Role of the Local DNS Server

Every host is configured with a local DNS server (provided by the ISP or organization).

  • Advantages:
    • Reduced Delay: Caching popular translations locally speeds up resolution for many users.
    • Reduced Load: It prevents repetitive queries from leaving the local network, saving external bandwidth.
  • Disadvantages:
    • Single Point of Failure: If the local DNS server is down, hosts on that network cannot resolve any names.
  • Mitigation: Redundant local DNS servers (primary and secondary) are used to ensure availability.

15.4 DNS Caching

Once a name server learns a mapping, it caches it locally.

  • Benefit: Greatly improves response time and reduces network traffic.
  • TTL (Time to Live): Cached entries disappear after a set time.
  • Issue: Cached entries can be out-of-date if a host changes its IP address before the TTL expires.

15.5 DNS Resource Records (RR)

The DNS database stores resource records (RR) with the format: (Name, Value, Type, TTL).

Type Name Value
A Hostname IP Address
NS Domain (e.g., foo.com) Hostname of authoritative name server
CNAME Alias name Canonical (real) name
MX Alias name Name of mail server associated with the name

15.6 DNS Protocol and Messages

  • Protocol: DNS primarily uses UDP on port 53 for queries and replies. TCP is used for large zone transfers between servers.
  • Message Format:
    • Identification: 16-bit # for the query; reply uses the same #.
    • Flags: Indicate if it's a query or reply, if recursion is desired, and if the reply is authoritative.
    • Questions: Name and type being queried.
    • Answers: RRs in response to the query.
    • Authority: Records for authoritative servers.
    • Additional: Other helpful info (e.g., the IP address of a name server mentioned in the Answer).

15.7 Diagnostic Tools

  • nslookup: Queries DNS servers manually.
  • dig: A more modern and flexible tool for DNS lookups (e.g., dig www.google.com +short).
  • whois: Used to find who owns an IP address block or domain name.

16. Video Streaming and Content Distribution Networks (CDNs)

16.1 Video Streaming Basics

Streaming video applications must overcome the challenges of variable network delays and packet loss to provide a smooth user experience.

  • Chunk: A small, self-contained segment of a larger video file (e.g., 2–10 seconds long). Chunks allow the client to switch bitrates mid-stream to adapt to changing bandwidth.
  • Continuous Playout Constraint: The client must play out the video at its original timing (e.g., 30 frames/sec), even if network delays fluctuate.
  • Jitter: Variability in the arrival time of packets.
  • Client-Side Buffering: The client stores a few seconds of video before starting playout. This buffer absorbs jitter and ensures continuous playout even if the network slows down temporarily.

16.2 DASH: Dynamic Adaptive Streaming over HTTP

DASH is the standard for modern video streaming (used by YouTube, Netflix).

  • Server-Side:
    • The video is divided into multiple chunks.
    • Each chunk is encoded at multiple different bitrates (qualities).
    • A Manifest File (Media Presentation Description - MPD) provides URLs for these different chunks and their available bitrates.
  • Client-Side Intelligence:
    • The client periodically estimates its available bandwidth.
    • It requests one chunk at a time, choosing the maximum encoding rate that its current bandwidth can support.
    • It can switch between qualities dynamically as network conditions change.

16.3 Content Distribution Networks (CDNs)

To serve millions of users simultaneously, a single "mega-server" is insufficient due to bandwidth limits, single points of failure, and high latency for distant users.

  • CDN Strategies:
    1. Enter Deep: Deploy small server clusters deep into thousands of access networks (ISPs) worldwide (e.g., Akamai).
    2. Bring Home: Build larger clusters in a smaller number of strategic locations (POPs) near access networks (e.g., Limelight).
  • CDN Operation:
    • Content is replicated across various CDN nodes.
    • When a user requests a video, the CDN must select a "good" node to stream from.
    • Selection Methods:
      • Geographic proximity.
      • Shortest delay/hop count (monitored via pings).
      • Client-side choice (giving the client a list of servers to test).

17. Socket Programming

A socket is the interface between the application layer and the transport layer (the "door" of the process).

17.1 Socket Identifiers

  • IP Address: Identifies the host (NIC).
  • Port Number: Identifies the specific process.
    • 1–255: Standard services (80: HTTP, 25: SMTP, 53: DNS).
    • 1–1023: Restricted to system/privileged use.
  • 5-Tuple: A unique network connection is defined by: (Source IP, Source Port, Destination IP, Destination Port, Protocol).

17.2 UDP Socket Programming (Unreliable Datagram)

  • Connectionless: No handshaking before sending data.
  • Explicit Addressing: The sender must attach the destination IP and Port to every packet.
  • Unreliable: Data may be lost or arrive out-of-order.
  • Demultiplexing: Based only on the destination port number.

17.3 TCP Socket Programming (Reliable Stream)

  • Connection-Oriented: Requires a "handshake" to establish a connection before data exchange.
  • Two Socket Types on Server:
    1. Welcoming Socket: Listens for incoming connection requests from any client.
    2. Connection Socket: Created by the server specifically for each client after the handshake, allowing the server to handle multiple simultaneous clients.
  • Reliable: Guarantees in-order, error-free delivery of a byte stream.
  • Demultiplexing: Based on the full 4-tuple (Source IP, Source Port, Dest IP, Dest Port).

17.4 Comparison Summary

Feature UDP Sockets TCP Sockets
Connection Connectionless Connection-oriented
Reliability Best-effort (unreliable) Reliable byte stream
Addressing Per-packet destination Set once during connect
Demux Criteria Dest Port 4-tuple (IPs and Ports)
API Complexity Simpler (sendto/recvfrom) More complex (listen/accept/connect)

18. Advanced Scenarios and Exam Calculations

18.1 Multi-Level DNS Resolution Delay

Scenario: A laptop queries for `www.xyz.com`. Local DNS does not have it cached.

  • Laptop to Local DNS (RTT): 6 ms (3 ms each way).
  • Local DNS to Root (RTT): 21 ms.
  • Local DNS to TLD (.com) (RTT): 9 ms.
  • Local DNS to Authoritative (dns.xyz.com) (RTT): 50 ms.

Calculation: The local DNS server must perform an iterated query:

  1. Query Root: +21 ms
  2. Query TLD: +9 ms
  3. Query Authoritative: +50 ms
  4. Total Local DNS delay: 80 ms.
  5. Add Laptop-to-Local delay: 6 ms. Total Delay: 86 ms.

18.2 DNS + HTTP Combined Latency

Scenario: User clicks a link. Host must perform DNS lookup first.

  • DNS RTTs: Local Cache (3 ms), 2nd server (28 ms), 3rd server (32 ms), 4th server (32 ms).
  • HTTP RTT (Local to Server): 9 ms.
  • Transmission Time: Negligible.

A. Base HTML Object Only (Non-persistent):

  • DNS Total: $3 + 28 + 32 + 32 = 95$ ms.
  • HTTP TCP Setup: 9 ms.
  • HTTP Request/Response: 9 ms.
  • Total: 113 ms.

B. Base + 3 Small Objects (Non-persistent, No Parallel):

  • Previous: 113 ms.
  • Each additional object: 2 RTTs (18 ms).
  • Total: $113 + (3 \times 18) = \mathbf{167}$ ms.

C. Base + 3 Small Objects (Non-persistent, 5 Parallel):

  • Base: 113 ms.
  • The 3 objects are requested in parallel over 3 new TCP connections.
  • Parallel delay: 2 RTTs (18 ms).
  • Total: 131 ms.

D. Base + 3 Small Objects (Persistent, 5 Parallel):

  • Base: 113 ms.
  • The 3 objects are requested over the existing connection.
  • Pipelining delay: 1 RTT (9 ms).
  • Total: 122 ms.

18.3 Playout Delay in Video Streaming

To calculate the minimum initial playout delay ($d$) such that no chunk misses its deadline:

  1. Identify the chunk with the largest network delay ($L_i = \text{Arrival}_i - \text{Transmission}_i$).
  2. The playout time of the first chunk ($P_1$) must be late enough that all subsequent chunks $i$ arrive at or before their scheduled playout time: $P_i = P_1 + (i-1) \times \text{Interval}$.
  3. Rule of Thumb: The playout delay must be at least as large as the maximum network delay experienced by any chunk.
