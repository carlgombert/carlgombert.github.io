Transport Layer

The transport layer provides logical communication between application processes running on different hosts. While the network layer provides communication between hosts, the transport layer extends this to the specific processes running on those hosts.

1. Transport Layer Overview

1.1 Responsibilities

  • Sender Side: Breaks application messages into smaller chunks called segments and passes them to the network layer.
  • Receiver Side: Reassembles segments into messages and passes them to the application layer.
  • Logical Communication: From the application's perspective, it's as if the processes are directly connected, even though they may be separated by many routers and links.

1.2 Transport vs. Network Layer

  • Network Layer: Logical communication between hosts. (e.g., IP protocol).
  • Transport Layer: Logical communication between processes. (e.g., TCP, UDP).

[!TIP] The Household Analogy Imagine two houses, each with 12 children who write letters to each other.

  • Children: Application processes.
  • Letters: Application messages.
  • Ann and Bill (the parents who collect/distribute mail): Transport layer protocols.
  • Postal Service: Network layer protocol. Ann and Bill only move mail from the children to the mailbox; the Postal Service moves mail from mailbox to mailbox.

2. Multiplexing and Demultiplexing

This is the process of extending host-to-host delivery to process-to-process delivery.

  • Multiplexing (at Sender): Gathering data from multiple sockets, enveloping data with headers (which will later be used for demultiplexing) to create segments, and passing them to the network layer.
  • Demultiplexing (at Receiver): Delivering the data in a transport-layer segment to the correct socket.

2.1 How Demultiplexing Works

Each transport-layer segment contains fields that allow the receiver to identify the target socket:

  1. Source Port Number
  2. Destination Port Number

UDP Demultiplexing (Connectionless)

  • A UDP socket is identified by a 2-tuple: (Destination IP, Destination Port).
  • When a host receives a UDP segment, it checks the destination port and directs it to the corresponding socket, regardless of where it came from (source IP and source port are ignored for demuxing purposes).

TCP Demultiplexing (Connection-Oriented)

  • A TCP socket is identified by a 4-tuple: (Source IP, Source Port, Destination IP, Destination Port).
  • The receiver uses all four values to direct the segment to the appropriate socket. This allows a server to support many simultaneous TCP sockets, each associated with a different connection, even if they share the same destination port (e.g., port 80).

3. Connectionless Transport: UDP

UDP is a "no-frills," lightweight transport protocol. It provides best-effort delivery with minimal overhead.

3.1 UDP Segment Format

The UDP header is only 8 bytes long, consisting of four 16-bit fields:

  • Source Port Number: For return traffic.
  • Destination Port Number: For demultiplexing at the receiver.
  • Length: Length (in bytes) of the entire UDP segment (header + data).
  • Checksum: Used for error detection.

3.2 UDP Checksum Calculation

The goal is to detect "flipped bits" in the segment.

  • Sender:
    1. Treats segment contents (including header and a "pseudo-header") as a sequence of 16-bit integers.
    2. Computes the one's complement sum of these integers.
    3. The result is stored in the Checksum field.
  • Receiver:
    1. Computes the sum of all 16-bit integers in the received segment (including the checksum field itself).
    2. If the result is all 1s (1111111111111111), no error is detected.
    3. If any bit is 0, an error occurred.

3.3 Why use UDP?

  • No Connection Setup: No delay from handshaking (important for DNS, VoIP).
  • Small Header Overhead: Only 8 bytes vs. 20 bytes for TCP.
  • No Congestion Control: UDP can blast data as fast as the application desires (useful for real-time apps where speed is more important than 100% reliability).

4. Principles of Reliable Data Transfer (rdt)

Reliable data transfer is one of the most important challenges in networking. The complexity of a reliable data transfer protocol depends strongly on the characteristics of the underlying unreliable channel.

4.1 rdt Service Model: Function Definitions

  • rdt_send(): Called by the application to pass data to the transport layer.
  • udt_send(): Called by the rdt protocol to transfer a packet over the unreliable channel.
  • rdt_rcv(): Called when a packet arrives at the receiver side.
  • deliver_data(): Called by the rdt protocol to deliver data to the application.

4.2 rdt1.0: Reliable Transfer over a Reliable Channel

In this idealized scenario, the underlying channel is perfectly reliable (no bit errors, no lost packets).

  • Sender: Simply makes a packet and sends it.
  • Receiver: Simply receives the packet and delivers the data.

4.3 rdt2.0: Channel with Bit Errors

In this scenario, bits in a packet may be flipped.

  • Error Detection: Uses a checksum to detect bit errors.
  • Feedback:
    • Acknowledgements (ACKs): Receiver tells the sender that the packet was received correctly.
    • Negative Acknowledgements (NAKs): Receiver tells the sender that the packet had errors.
  • Retransmission: The sender retransmits the packet upon receiving a NAK.

[!IMPORTANT] Stop-and-Wait Protocol In rdt2.0, the sender sends one packet and then waits for the receiver's response (ACK or NAK) before sending the next packet. This ensures the receiver is ready and has successfully received the current data.

The Flaw in rdt2.0

What happens if the ACK or NAK itself is corrupted?

  1. The sender doesn't know if the receiver got the data correctly.
  2. The sender cannot simply retransmit because it might create a duplicate at the receiver (if the first packet was actually received OK).

4.4 rdt2.1: Handling Garbled ACKs/NAKs (Sequence Numbers)

To solve the duplicate problem, rdt2.1 introduces Sequence Numbers.

  • Sender Side:
    • Adds a sequence number (0 or 1) to each packet.
    • If an ACK/NAK is corrupted or is a NAK, the sender retransmits the current packet.
    • Two sequence numbers (0 and 1) are sufficient for a Stop-and-Wait protocol because the sender only ever needs to distinguish the current packet from the immediately preceding one.
  • Receiver Side:
    • Checks the sequence number of arriving packets.
    • If it receives a packet with a sequence number it already processed (a duplicate), it discards the data but sends an ACK to the sender (to ensure the sender finally moves on).

4.5 rdt2.2: A NAK-Free Protocol

rdt2.2 achieves the same functionality as rdt2.1 but uses ACKs only (no NAKs).

  • Mechanism:
    • Instead of sending a NAK for a corrupted packet, the receiver sends an ACK for the last correctly received packet.
    • The receiver must explicitly include the Sequence Number of the packet being ACKed.
  • Sender Action:
    • A duplicate ACK (an ACK for a packet the sender already knows was received) is treated like a NAK. It indicates that the next packet was likely corrupted, and the sender retransmits the current packet.
    • Note: This approach is used by TCP.

5. rdt3.0: Channel with Errors and Loss

In a real-world network, packets can not only be corrupted but also lost. rdt3.0 handles both bit errors and packet loss.

5.1 Mechanics of rdt3.0

  • Timer: The sender starts a timer after sending a packet. If an ACK doesn't arrive within a certain "timeout" period, the sender assumes the packet (or its ACK) was lost and retransmits.
  • Sequence Numbers: As in rdt2.1, sequence numbers (0, 1) are used to handle potential duplicate packets caused by premature timeouts.
  • Alternative Bit Protocol: Because it uses a 1-bit sequence number (0 or 1), rdt3.0 is often called the Alternative Bit Protocol.

6. Pipelined Reliable Data Transfer

The "Stop-and-Wait" approach of rdt3.0 is highly inefficient because it only allows one packet to be "in-flight" at a time.

6.1 Performance Analysis of Stop-and-Wait

Utilization ({sender}$) is the fraction of time the sender is actually busy sending bits into the link.

  • Formula: 68413U_{sender} = \frac{L/R}{RTT + L/R}68413
    • $: Packet length in bits.
    • $: Transmission rate (bps).
    • $: Round-Trip Time.
  • Inefficiency: In networks with a large delay-bandwidth product (high speed, long distance), the utilization is extremely low (often < 1%). The network protocol, not the physical hardware, becomes the bottleneck.

6.2 Pipelining Solution

Pipelining allows the sender to send multiple packets without waiting for an ACK. This increases utilization by a factor of $ (the number of in-flight packets). 68413U_{sender} = \frac{N \times L/R}{RTT + L/R}68413

There are two major forms of pipelined protocols:

6.2.1 Go-Back-N (GBN)

  • Sender:
    • Has a "window" of up to $ consecutive unacknowledged packets.
    • Uses Cumulative ACKs: An ACK for sequence number $ indicates that all packets up to and including $ have been received correctly.
    • Has one timer for the oldest unacknowledged packet.
    • On Timeout: Retransmits all unacknowledged packets in the window (Goes back to $).
  • Receiver:
    • Only accepts packets in-order.
    • Discards out-of-order packets (does not buffer them).
    • Sends an ACK for the last correctly received in-order packet.

6.2.2 Selective Repeat (SR)

  • Sender:
    • Maintains a window of $ packets.
    • Uses Individual ACKs for each packet.
    • Maintains a timer for each unacknowledged packet.
    • On Timeout: Retransmits only the specific packet that timed out.
  • Receiver:
    • Buffers out-of-order packets until missing packets arrive to fill the gap.
    • Acknowledges each packet individually.

6.3 Sequence Number Space and Window Size

To avoid "packet confusion" (where the receiver can't tell if a packet is a new one or a retransmission from a previous window wrap-around), the following rules must be followed:

  • For GBN: \le S - 1$ (where $ is the total sequence number space).
  • For SR: \le S / 2$.

7. TCP: Transmission Control Protocol

TCP is the most widely used transport protocol for reliable data transfer on the Internet.

7.1 TCP Overview

  • Point-to-Point: One sender, one receiver.
  • Reliable, In-Order Byte Stream: No "message boundaries"; the application sees a continuous stream of bytes.
  • Full Duplex: Bi-directional data flow in the same connection.
  • Connection-Oriented: Requires a three-way handshake before data transfer.
  • Pipelined: Uses windows to allow multiple in-flight segments.

7.2 TCP Segment Structure

A TCP segment consists of a header (typically 20 bytes) and a data field.

  • Source/Dest Port: For multiplexing/demultiplexing.
  • Sequence Number: The byte-stream number of the first byte in the segment's data.
  • Acknowledgement Number: The sequence number of the next byte expected from the other side.
  • Flags:
    • ACK: Indicates the ACK field is valid.
    • SYN/FIN: Used for connection setup and teardown.
    • RST: Reset the connection.
    • PSH/URG: Push data immediately / Urgent data (rarely used).
  • Receive Window (rwnd): Used for flow control; tells the sender how many bytes the receiver is willing to accept.
  • Checksum: Internet checksum for error detection.

7.3 TCP Reliability and ACK Generation

TCP is essentially a hybrid of GBN and SR, but closer to GBN with some enhancements.

Feature TCP Approach
ACK Type Cumulative ACKs (like GBN).
Timers Single timer for the oldest unacknowledged segment.
Buffering Receiver usually buffers out-of-order segments (like SR).
Retransmission On timeout, retransmits only the oldest unacknowledged segment.

ACK Generation Rules (RFC 5681)

  1. Delayed ACK: If an in-order segment arrives and everything is already ACKed, wait up to 500ms for the next segment before sending an ACK.
  2. Cumulative ACK: If an in-order segment arrives and another is pending, send one cumulative ACK immediately.
  3. Duplicate ACK: If an out-of-order segment arrives (gap detected), immediately send a duplicate ACK for the next expected byte.
  4. Gap Fill: If a segment arrives that partially or completely fills a gap, send an ACK immediately.

9. Header Overhead and Protocol Efficiency

The choice of transport protocol (TCP vs. UDP) and the size of the payload significantly impact the efficiency of network transmissions.

9.1 Header Sizes

  • UDP + IP: 8 bytes (UDP) + 20 bytes (IP) = 28 bytes of header.
  • TCP + IP: $\ge$ 20 bytes (TCP) + 20 bytes (IP) = $\ge$ 40 bytes of header.

9.2 Overhead Scenarios

Overhead is calculated as: $\frac{\text{Total Header Size}}{\text{Total Packet Size}}$.

  1. Full MSS (Maximum Segment Size, ~1500 bytes):

    • TCP/IP Overhead: $\ge 40 / 1500 \approx \mathbf{2.67%}$.
    • UDP/IP Overhead: / 1500 \approx \mathbf{1.87%}$.
    • Conclusion: For large data transfers, the overhead difference is negligible.

  2. Single Byte Payload (e.g., Telnet keystroke):

    • TCP/IP Overhead: $\ge 40 / (40 + 1) \approx \mathbf{97.6%}$.
    • UDP/IP Overhead: / (28 + 1) \approx \mathbf{96.6%}$.
    • Conclusion: For very small messages, the transmission is almost entirely headers. This is a common inefficiency in real-time or interactive applications.

10. TCP Congestion Control

Congestion occurs when there are too many sources sending too much data too fast for the network to handle. This is distinct from Flow Control, which protects a single receiver from being overwhelmed by a single sender.

10.1 Approaches to Congestion Control

  1. End-to-End Congestion Control: The network provides no explicit feedback. Congestion is inferred by the end systems based on observed packet loss and delay. (This is the approach taken by TCP).
  2. Network-Assisted Congestion Control: Routers provide direct feedback to the sender (e.g., using a congestion bit like ECN).

10.2 TCP Congestion Control: AIMD

TCP uses the Additive Increase Multiplicative Decrease (AIMD) algorithm to probe for available bandwidth.

  • Additive Increase: Increase the congestion window (cwnd) by 1 MSS every RTT until a loss event occurs.
  • Multiplicative Decrease: Cut cwnd in half after a non-timeout loss event (triple duplicate ACKs).
  • Sawtooth Behavior: The cwnd grows linearly and then drops sharply, creating a "sawtooth" pattern as it probes for the network's capacity.

10.3 Congestion Control Phases

TCP manages its sending rate using two main variables: cwnd (congestion window) and ssthresh (slow start threshold).

1. Slow Start (SS)

  • Initial State: cwnd = 1 MSS.
  • Growth: cwnd doubles every RTT (exponential growth). For every ACK received, cwnd increases by 1 MSS.
  • Purpose: Rapidly find the available bandwidth.
  • Transition: When cwnd >= ssthresh, TCP switches to Congestion Avoidance.

2. Congestion Avoidance (CA)

  • Growth: cwnd increases by 1 MSS every RTT (linear growth).
  • Purpose: Carefully probe for additional bandwidth without overwhelming the network.

10.4 Reacting to Loss Events

TCP reacts differently depending on the type of loss detected:

Event Reaction (TCP Tahoe) Reaction (TCP Reno)
Timeout (Severe) ssthresh = cwnd / 2; cwnd = 1; Enter Slow Start. ssthresh = cwnd / 2; cwnd = 1; Enter Slow Start.
Triple Duplicate ACKs ssthresh = cwnd / 2; cwnd = 1; Enter Slow Start. ssthresh = cwnd / 2; cwnd = ssthresh + 3; Enter CA (Fast Recovery).

[!IMPORTANT] Fast Retransmit and Fast Recovery TCP Reno's ability to stay in Congestion Avoidance after triple duplicate ACKs is called Fast Recovery. It assumes that receiving triple duplicate ACKs indicates the network is still capable of delivering some packets, whereas a timeout suggests more severe congestion.

10.5 TCP Throughput

The average throughput of a TCP connection can be approximated as: 68459\text{Throughput} \approx \frac{1.22 \times MSS}{RTT \times \sqrt{L}}68459 (where $ is the loss rate). Roughly, the rate is $\frac{cwnd}{RTT}$ bytes/sec.

11. TCP Timeout and RTT Estimation

TCP uses a timeout mechanism to trigger retransmissions. Setting the timeout value correctly is crucial: too short causes unnecessary retransmissions; too long results in slow reaction to loss.

11.1 Estimating Round-Trip Time (RTT)

  • SampleRTT: The measured time from when a segment is sent until its ACK is received.
  • EstimatedRTT: An exponential weighted moving average (EWMA) of SampleRTTs. It smooths out fluctuations.
    • Formula: EstimatedRTT = (1 - \alpha) * EstimatedRTT + \alpha * SampleRTT (typically $\alpha = 0.125$).
  • DevRTT: An estimate of how much SampleRTT typically deviates from EstimatedRTT.
    • Formula: DevRTT = (1 - \beta) * DevRTT + \beta * |SampleRTT - EstimatedRTT| (typically $\beta = 0.25$).

11.2 Calculating Timeout Interval

TCP sets its timeout interval based on the estimated RTT and its variability: 68464\text{TimeoutInterval} = \text{EstimatedRTT} + 4 \times \text{DevRTT}68464 If a timeout occurs, the TimeoutInterval is doubled to avoid overwhelming a potentially congested network. It is reset once a new SampleRTT is measured.

12. TCP Flow Control

While Congestion Control prevents the sender from overwhelming the network, Flow Control prevents the sender from overwhelming the receiver's buffer.

12.1 The Receive Window (rwnd)

The receiver maintains a receive buffer (RcvBuffer). The application might be slow to read from this buffer.

  • rwnd: The amount of free space remaining in the receive buffer.
  • Mechanism: The receiver includes the current rwnd value in the header of every segment it sends back to the sender.
  • Sender Constraint: The sender ensures that the amount of unacknowledged ("in-flight") data is always less than or equal to the receiver's advertised rwnd.
    • LastByteSent - LastByteAcked \le rwnd

13. TCP Connection Management

13.1 Three-Way Handshake (Connection Setup)

Before exchanging data, the client and server must establish a connection:

  1. SYN: Client sends a segment with the SYN bit set and an initial sequence number (client_isn).
  2. SYN-ACK: Server allocates buffers, sets the SYN and ACK bits, sets the ACK number to client_isn + 1, and chooses its own initial sequence number (server_isn).
  3. ACK: Client responds with an ACK bit set and the ACK number server_isn + 1. This segment may also contain application data.

13.2 Closing a Connection (Four-Way Handshake)

  1. FIN: Client sends a segment with the FIN bit set.
  2. ACK: Server responds with an ACK.
  3. FIN: When the server is ready to close, it sends its own FIN segment.
  4. ACK: Client responds with an ACK and enters a TIMED_WAIT state.
    • TIMED_WAIT: The client waits (typically for 2 * Maximum Segment Lifetime) before fully closing. This ensures that the last ACK was received and handles any "stray" segments still in the network.

14. Modern Evolution: QUIC and HTTP/3

As web applications become more complex, traditional TCP has shown limitations, specifically in setup delay and Head-of-Line (HOL) blocking.

14.1 QUIC (Quick UDP Internet Connections)

Initially developed by Google and now an IETF standard (RFC 9000), QUIC is a transport-layer protocol implemented on top of UDP.

  • Combined Handshake: QUIC combines the transport and security (TLS 1.3) handshakes into a single exchange, reducing connection setup to 1 RTT (compared to 2-3 RTTs for TCP+TLS).
  • No Head-of-Line (HOL) Blocking:
    • In TCP, if one packet is lost, the entire byte stream is halted until it is retransmitted (HOL blocking).
    • QUIC supports multiple independent streams within a single connection. If a packet for one stream is lost, other streams can continue to deliver data to the application.
  • Connection Migration: QUIC uses connection IDs rather than 4-tuples, allowing a connection to stay alive even if a user's IP address changes (e.g., switching from WiFi to 4G).

14.2 HTTP/3

HTTP/3 is the latest version of the Hypertext Transfer Protocol, and it runs exclusively over QUIC. This shift allows the Web to benefit from QUIC's faster setup and improved performance in lossy environments.
